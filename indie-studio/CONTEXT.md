# AI 開発ハーネス

個人開発者が「企画 → プロトタイプ → 設計 → 実装 → リリース」までを一人で回すための、AI を駆使した開発ハーネス（制御層）の文脈。このリポジトリ（`claude-collections`）の `indie-studio` コレクションはハーネス自体の設計判断とドメイン語彙を保持する。個々のサービス（例: socialcoffeenote）の企画・設計成果物はここには置かない。

## Language

**ハーネス（Harness）**:
モデルを駆動する足場一式。ステージ群・エージェント・成果物の受け渡し・「いつ人間に聞くか」の判断を束ねる制御層を指す。
_Avoid_: パイプライン（ステージの並びだけを指す狭義の語として区別する）、ワークフロー

**ステージ（Stage）**:
ハーネスを構成する工程の単位（企画 / プロトタイプ / 技術設計 / 分解 / 実装 など）。各ステージは前ステージの成果物を入力に取る。

**アンカー（Anchor）**:
人間が対話で決める土台。**人間の意思決定境界**であり、これより下流はアンカーから導出される。集合は4点：① PRFAQ（価値とゴール）② デザインプリンシプル（優先順位）③ 提供形態（iOS アプリ／Web 等）④ マネタイズするかどうか。ペルソナ・使用シーンはアンカーに含めず、PRFAQ から自律導出する。
_Avoid_: 前提、ベース

**導出ドキュメント（Derived doc）**:
アンカーが固まった後、エージェントが**自律的に**調査・生成する派生成果物（ペルソナ・競合・機能スコープ・画面仕様など）。人間が一問一答で共著するものではない。
_Avoid_: 設計書（実行モードの区別が消えるため避ける）

**プロトタイプ（Prototype）**:
導出ドキュメント群からエージェントが起こす、プロトタイプゲートでレビューする対象の成果物。人間はこれを起点に「なぜこの形か」を問い、根拠となる導出ドキュメントを辿る。

**レビューゲート（Review gate）**:
各自律フェーズの直後に置かれる人間のレビュー関所。レビューは**フェーズごとに分割**され、各ゲートはそのフェーズの主成果物を、**フェーズ固有の観点**で見る（プロトタイプゲート＝体験・UX、技術設計ゲート＝データモデル・アーキテクチャ）。あるフェーズの観点を別フェーズのゲートに持ち込まない。ゲートはレビューだけでなく**繰り越し決定**を確定させる場でもある。

**繰り越し決定（Deferred decision）**:
触ってからの方が決めやすい判断を、エージェントに解決させず、プロトタイプに**未決の論点**として持ち込み、レビューゲートで人間が確定させるもの。アンカー（事前に握る）でも導出（エージェントが解決）でもない第三のカテゴリ。第1号はマネタイズの無料/有料境界。機構：所有ページに **⚠️繰り越し マーカー＋候補**を inline で残し、ディレクターがゲートで走査・提示する（専用の決定ログ file は持たない・ADR-0019）。

**Claude Design**:
Anthropic の AI デザインツール。プロンプトから HTML/CSS/JS の対話的プロトタイプを生成し、自然言語・インライン編集・スライダーで refine できる。Claude Code へハンドオフバンドルを渡す。プロトタイプ段の**媒体兼プロトタイプゲートの操作面**。フロントエンド専用（バックエンド生成なし）。

**ハンドオフバンドル（Handoff bundle）**:
Claude Design が出力し Claude Code が消費する**一方向**（design → code）のパッケージ。HTML/CSS/JS ＋各状態スクショ＋コンポーネント spec（機械可読）＋デザイントークン＋README（stack・パス・DoD）＋各ページの意図を含む。これが**画面仕様＝正当化トレイルを兼ねる**ため、画面仕様を先に手書きする必要が消える。

**プロトタイプブリーフ（Prototype brief）**:
アンカーから自律 deriver が生成する、Claude Design に渡すプロンプト/入力。ユーザーが「速く作りたい」と言った“プロトタイプのデザインを作るプロンプト”の正体であり、ハーネスの新しい中核部品。

**ハーネスエンジニアリング（Harness engineering）**:
**オーケストレーター（制御層）**と**各ステージの自走エージェント**を、機械可読な契約と人間ゲートのもとに工学する営み。到達状態は「アンカー4点を入力し、各レビューゲートで承認するだけでリリース可能なソフトウェアに至る」こと。人間の介入をアンカー＋ゲート＋繰り越し決定に局所化し、ステージ間の手配線（指揮者業務）をゼロにすることを目的とする。現状の binding constraint は制御層の不在（a）と各エージェントの自走品質（b）の**両方**。

**オーケストレーター（Orchestrator／制御層）**:
ステージを自律連結し、契約に従って次段を起動し、ゲートでのみ人間を呼ぶ制御層。`claude-collections/indie-studio` が実装すべき本体。現状は**不在**で、人間がその役を手で担っている（＝指揮者業務）。
_Avoid_: ランナー、ドライバ

**大枠ゲート（Framework gate）**:
人間が「枠組みを決める／検証する」ために介入する数少ない関所。全5つ（G1 企画/アンカー・G2 プロトタイプ・G3 アーキ/インフラ・G4 分解リスト・G5 PR レビュー+merge／ADR-0007）。G1〜G4 は決める、G5 は検証する。これ以外の判断・操作（起票・精緻化・push・PR open）は自律。原則：**枠組みがズレれば細部の正しさは無意味**。

**自走設計（Autonomous-by-design）**:
エージェントが内部の曖昧点で停止して人間に聞くのではなく、根拠ある決定を**下し・記録し・進み**、設計されたゲートでのみ人間に晒す設計思想。対義は**補助設計**（停止して聞く co-pilot）。harness engineering の中核原則。

**self-grill（自己 grill）**:
エージェントが griller と answerer の両役を演じ、anchors／導出ドキュメント／プロトタイプ／コードを**答え合わせ材料（corpus）**に自答して要件を精緻化する自走精緻化。人間対話の grill を置き換える。「context から既約 **かつ** 不可逆/高ステークス」の決定だけをゲートで晒す（決定単位のランタイム判断＝ティアリング方式 Y を採用）。

**導出エージェント（Deriver agent）**:
企画→ブリーフ段で、担当する corpus 成果物を self-grill で自律導出する専門エージェント。成果物名ではなく**現実の職種**に束ねる。S1 の実在職種ロスター（UX リサーチャー・プロダクトマネージャー・ビジネスストラテジスト・プロダクトデザイナー）は ADR-0022。ディレクター（オーケストレーター）が依存順に起動する。
_Avoid_: ステージワーカー（feature-team の実装 worker と混同しない）

**評価エージェント／レビュアー（Reviewer / Critique agent）**:
導出エージェント（＝[[担当職種]]）の成果物を、anchors・真実源・カバレッジ逆引きで self-grill 評価し、満たさなければ担当職種へ差し戻す職種。S1 の評価役は「レビュアー」（ADR-0022・「プロダクトクリティーク」は活動であって職種でないため改称）。self-grill の griller 役を独立 context の役者として立てたもの。差し戻し protocol（カデンツ・予算・カスケード・起動形）は ADR-0018。

**担当職種（Responsible role）**:
ある成果物を導出/実装した職種エージェント（例：screen-specs ならプロダクトデザイナー、persona なら UX リサーチャー）。差し戻し時に continuation 再起動されて自分の成果物を直す主体。**ディレクター（オーケストレーター）は corpus を自ら書かず**、担当職種の起動・findings のルーティング・完全性ガードへの集計を担う点で担当職種と別物。
_Avoid_: プロデューサ（非正規語・ディレクターと混同を招く）

**finding（差し戻し finding）**:
プロダクトクリティークが評価で出す差し戻しの最小単位。担当職種が ✅解消／➖省略(理由)／⚠️未達(理由) の3状態で決着させ、ディレクターが完全性ガードに集計する。構造（対象・観点・重大度・根拠・期待・提案）は ADR-0018。

**デザイントークン（Design token）**:
色・タイポ・余白・角丸・モーション等のデザイン変数群。デザインシステムエンジニアが起こす具体値で、**Claude Design へ渡して見た目を準拠させ統一プロトを作る**ための拠り所。別ファイル（.css/.json）は持たず、repo-root の [[design-md]] のカラー/タイポ等セクションに具体値として載る（ADR-0020）。

**DESIGN.md（デザイン憲法）**:
デザインシステムを1枚に集約した repo-root の「見た目の憲法」。**意図ではなく具体値＋禁止事項**で書き（AI は具体に従いやすい）、Claude Code・Claude Design 等の全 AI エージェントが自動参照する standing/living file。プロトに先立って具体の拠り所を与えて Claude Design に統一プロトを作らせ、G2 以降 living に更新される。9セクション（ビジュアルテーマ/カラー/タイポ/コンポーネント/レイアウト/深さ/Do's&Don'ts/レスポンシブ/エージェント向けプロンプトガイド）。意図版 design-system.md は廃止。生成機構（誰が組み上げるか・mood は画像つき人間対話）は未確定（ADR-0020）。
_Avoid_: 意図版デザインシステム（DESIGN.md は realized でも intent でもなく concrete な憲法）

**画面一覧（Screen inventory）**:
機能スコープから導く「どの画面を作るか」の骨格。プロトタイプの**枠組み**であり、エージェントがドラフトを出し**人間が枠組みレビュー**（G4 相当の軽い関所）してから下流（screen-specs・ブリーフ）の自律導出に進む（ADR-0011）。各画面の詳細＝**screen-specs**（先行導出する期待値、G2 で育つ）。

## Naming convention

**サブステージ（Sub-stage）**:
main stage（`S1`〜`S5`）の中に挟まる autonomous なサブ工程の番号付け規約（ADR-0025）。サブステージは**英字 suffix**（`S1a`, `S1b`, ...）で命名し、**alphabetic 順 = 実行順**を不変条件とする。main stages（`S1`〜`S5`）と gates（`G1`〜`G5`）は不変。新規サブステージは末尾の英字を進めて命名し、既存サブステージ間に挿入する場合は後続の英字を順送り rename する。
_Avoid_: 小数点表記（`S1.5` 等）、numeric subindex（`S1-1` 等）。

## Flagged ambiguities

- 設計の幹（アンカーゲート〜5大枠ゲート〜自走設計〜self-grill〜Claude Design〜repo-native corpus）は **ADR-0001〜0010 で確定**。企画→ブリーフ段の編成（多職種エージェント8体）は **ADR-0011 で確定**。ドキュメントの2モードと vault 退役は **ADR-0012 で確定**。出力先は repo の `docs/discovery/{anchors,planning,design,brief.md,DECISIONS.md}`。残るは実装レベルの論点（fleet 廃止の移行手順・オーケストレーターの実装・信頼度ダイヤルの運用基準・既存薄実装の作り直し）→ `docs/ROADMAP.md` 参照。

## Findings

- **feature-team 失敗モード = (1) 入力欠乏 ＋ (2) 自走品質**。(3) 実行環境・(4) オーケストレーションは除外＝親の指揮と 12 体の並列起動そのものは回る。
- **根本原因（ユーザー再フレーム）= そもそも自走するための設計で作っていない**。既存スキルは「曖昧点で停止して人間に聞く」補助設計（feature-team の Phase 0/3/3.5 escalation が証拠）。harness engineering の本質は全エージェントを自走設計に作り替えること。2a/2b の粒度論より上位の問題。
- **上流フロー確定（ユーザー）**: アンカー（PRFAQ・デザインプリンシプルを人間対話）→ オーケストレーターが適切なエージェントを起動し service-designer/prototype-designer 出力を自律生成＋Claude Design 用ブリーフ生成 → Claude Design でプロトタイプ → **最初の HITL＝プロトタイプゲート**（人間が反復レビュー）。tech-designer も自律導出である点は原則上確定（明示確認は未）。
- **逆伝播＝人間駆動の並列デュアル訂正（ADR-0006）**: プロトタイプゲートでは Claude Design と Claude Code を並列稼働させ、人間が docs 修正（Claude Code）とプロトタイプ修正（Claude Design）を同時に出す。システム自動同期ではない。
- **プロトタイプ品質バー（端折り禁止）**: 自律導出は完全な corpus を生成する。「minimal」は人間の入力・対話の最小化を指し、導出物の最小化ではない。お粗末なプロトタイプは許容しない。質は対話ではなく self-grill の徹底度から出る。なお **screen-specs（画面仕様）は「先行導出する期待値」**であり、Claude Design に渡して無駄画面を防ぎ、G2 でプロトタイプとの差を訂正して育てる正当化トレイル（ADR-0011／0003／0006）。デザイン層の出口は moodboard ではなく **design-system ＋ design-tokens**（ADR-0011）。
- **企画→ブリーフ段＝多職種エージェント編成（ADR-0011）**: この段は単一スキルではなく、ディレクター（オーケストレーター）＋専門6（UXリサーチャー／PM／事業・マーケ／リスク・法務・品質／プロダクトデザイナー／デザインシステムエンジニア）＋評価1（プロダクトクリティーク）の8体。対話で作っていた service-designer＋prototype-designer の corpus を対話なしで同じ深さで生成する。画面一覧はドラフト後に人間が枠組みレビュー（G4 相当の軽い関所）。
- **ドキュメントは AI が書く・2モードのみ（ADR-0012）**: ハーネスの全文書は (1) AI が自律的に書く、(2) 人間との対話から AI が書く、の2種だけ。**人間は手で文書を書かない**。アンカーは (2)＝人間と対話して AI が書く（決めるのは人間・書くのは AI）。それ以外の導出 corpus は (1)＝自律。ゆえに Obsidian vault（人間の執筆面）は退役し、anchors も corpus も repo-native（`docs/discovery/anchors/` 等）。
- **共通ステージ形・ステージ間フィードバック（ADR-0013）**: 5ゲート（G1〜G5）のステージ列＝G1 アンカー / S1 企画→ブリーフ / S2 プロトタイプ / S3 技術設計 / S4 分解 / S5 実装。**S2（Claude Design＝外部）以外の全ステージは共通形「ディレクター（スキルを読んだメインセッションの制御役）＋職種エージェント群＋評価エージェント」**で corpus を生む。各ステージは**自律導出＋枠組み判断の対話の混合**（純・自律のステージは無い）。**feature-team / fleet は廃止**し S5 を共通形で作り直す。パイプラインは前方一方向でなく、**G2→S1（プロトタイプの明確化を screen-specs 核ドキュメントへ書き戻し）・S3→S1（技術見積もりで価格/NFR/実現可否を確定）**のフィードバックを持つ。
- **G5 適応ゲート（ADR-0008）**: レビュー要否は G4 で人間がチケット単位にタグ付け。非根幹は green で自動 merge、根幹のみ人間レビュー＋merge。精度向上に応じ自動 merge ゾーンを手で広げる。
- **agents＝ハーネスのホーム（ADR-0009）**: スキル・エージェント・オーケストレーター・設計を agents に集約。dotfiles の fleet は廃止。リポジトリ名は変更しうる。
- **corpus は repo-native（ADR-0010）**: corpus of record は各サービス repo の `docs/` サブツリー。Obsidian はアンカーの人間執筆面（任意）。コミット履歴の混在は commit scope で分離可能にする。横断性と原子性が採用理由。
