# decomposition

AI 自律開発ハーネスの **ステージ4（分解）** スキル。実行環境は **Claude Code**。ディレクターが職種エージェントを起動し、機能を**垂直スライス（=1PR）**に分解して `index.md`（骨格）を作り、**人間の承認（G4）後に issue トラッカー（Linear）へ起票**する。共通ステージ形（ADR-0013）。

到達点：**技術設計 → 起票済みの実装チケット群**を、人間の枠組みレビュー（index.md・G4）と起票承認だけで、F-ID カバレッジ漏れゼロで作る。

## いつ使うか

- 技術設計（tech-design）が固まり、実装単位に割る段階に入ったとき。
- F-ID 機能一覧・モジュール構造から issue を起こしたいとき。
- ハーネス4スキル（ADR-0017）の3番目として人間が起動するとき。

**ここで扱わないこと**：技術設計（上流 tech-design）。実装（下流スキル）。

## 入力（必須＝F-ID 機能一覧・モジュール）

- **S3 技術設計**：`docs/tech/`（接頭辞付き機能一覧 `F-{MODULE}-{連番}`・モジュール構造・ドメインモデル）。**F-ID 機能一覧とモジュールが欠ければ停止**（上流へ差し戻す）。
- **S1 screen-specs**：`docs/discovery/design/screen-specs/`（受入条件の材料）。
- **プロトタイプ実体**：参照。

## ロスターと依存順

| エージェント | 担当 |
|---|---|
| `engineering-manager` | 垂直スライス分解（=1PR）・依存順・capability 束ね（束ね親）・HITL/AFK 判定・レビュー要否タグ |
| `qa-engineer` | 各スライスの受入条件（BDD/チェックリスト） |
| `principal-engineer` | 評価（分解レビュー・差し戻し・ADR-0018） |

依存順：EM がスライス分解・依存・束ね → QA が受入条件 → principal-engineer が評価。**器構築 issue を依存の根**に置く。

## ディレクター制御フロー

スキル1/2 と同型。Agent tool で spawn、**ADR-0018 の評価ループ**、**完全性ガード（F-ID カバレッジ漏れゼロ）**。ゲートで**ギャップレポート＋繰り越し一覧**を提示。

## 出力と起票（2段）

1. **index.md（repo-native）**：`docs/decomposition/index.md`。垂直スライス＝1PR の骨格、依存順、capability 束ね、各スライスの F-ID 対応、HITL/AFK・レビュー要否タグ。**これが G4 レビューの対象**。
2. **〔G4: 人間が index.md をレビュー＋起票を明示承認〕**：粒度・依存・タグ・束ね境界を見る。**起票は外部書き込みなので省略不可の明示承認**。
3. **Linear へ起票**（承認後・自律）：issue トラッカーは外部でよい（repo-native 原則は文書 corpus に限る・ADR-0015）。issue 本文＝受入条件・参照リンク（screen-specs/tech docs）・冪等キー `S-{nn}`。

## タグ（2レイヤ）

- **HITL/AFK**：self-grill への粗いヒントとして残す（ADR-0005・完全廃止しない）。
- **レビュー要否（根幹/非根幹）**：G4 で付け、G5 の自動 merge/人間 merge を分ける（ADR-0008）。

## 対話点

index.md レビュー（粒度・依存・タグ・束ね境界＝G4）＋ 起票の明示承認。決めるのは人間、書くのは AI。

## self-grill・decide-record-proceed・繰り越し

スキル1/2 と同じ（ADR-0004/0005/0019）。根拠は inline、専用の決定ログ file は作らない。

## 破壊的操作の扱い

push / PR / merge / 課金 / 外部送信はしない。**例外：Linear 起票は G4 の明示承認後のみ**（外部書き込み・ADR-0007/0008）。

## 関連 ADR
