# G5 レビューの要否は G4 でチケット単位に決める／非根幹は自動 merge（適応ゲート）

PR レビュー（G5）を全件固定にせず、レビューの要否を **G4（分解リストレビュー）で人間がチケット単位にタグ付け**する。根幹（セキュリティ・課金・PII・秘密情報・不可逆マイグレーション・公開 API・インフラ）に触れるチケットは「要レビュー＋人間 merge」、非根幹は「green で自動 merge」。プロジェクトの精度が上がるほど、自動 merge ゾーンを人間が手で広げる（適応）。溜まった質の負債はリファクタチケットで一括返済する。

## Status

accepted（ADR-0007 の G5「merge は常に人間の不可逆操作」を改訂）

## Considered Options

- **却下: PR 時点で実装エージェントが critical/非 critical を自己認定**。利益相反でありセキュリティ回帰がすり抜ける最大の穴。判定はコードを書く前に人間が G4 で行う方が安全。
- **却下: 全 PR 固定レビュー**。序盤は妥当だが、精度が上がった後は人間の注意を浪費する。

## Consequences

- **非根幹チケットは自動 merge**。merge が人間ゲートになるのは根幹チケットのみ。
- **安全網（推奨）**：要レビュー外のチケットでも、実装が critical surface（パス/モジュールで保守的に機械判定）に触れたら自動でレビューへ昇格。境界は「疑わしきはレビュー」。
- G4 のリストレビューに「レビュー要否タグ」が加わる（既存 HITL/AFK ラベルの正統な後継）。
- 信頼度ダイヤル（自動 merge ゾーンの拡大）は人間が手動で広げる。メトリクスで自動的に緩めない（緩める判断こそ高ステークス）。
