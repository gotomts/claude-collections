# service-derivation

AI 自律開発ハーネスの **上流ループ自走スキル**。アンカー(人間が握る土台)を入力に、プロトタイプを起こすのに必要な corpus を **self-grill で完全に導出**し、Claude Design に渡す **プロトタイプブリーフ** を生成する。実行環境は **Claude Code**（カレントのサービス repo 上で起動）。

このスキルの到達点は「**既存アンカー → 触れるプロトタイプ** が、読める決定ログ付きで、人間の一問一答ゼロで到達できる」こと。質は対話の量ではなく **self-grill の徹底度** から出す。お粗末なプロトタイプは成果物として認めない。

> 出自: 本スキルの運用規則は harness の設計判断 ADR-0001〜0010（`gotomts/agents` repo）に基づく。ただし本スキルは任意のサービス repo で単体起動できるよう、運用規則はすべて本ファイルにインライン化してある（実行時に agents repo の存在を前提にしない）。

## ハーネス全体の中での位置

```
アンカー(人間/G1)
  → 〔このスキル〕self-grill 導出 → 完全な corpus ＋ Claude Design ブリーフ
  → Claude Design でプロトタイプ → プロトタイプゲート(人間/G2・並列デュアル訂正)
  → tech-designer(技術設計) → G3 → 分解 → G4 → feature-team(実装) → G5
```

- **置換対象**: 旧 `service-designer` の対話部分（一問一答の共著）と、廃止した `prototype-designer`（先行画面仕様の手書き）/ `prototype-builder`（HTML 手起こし）。これらが socialcoffeenote で「3 週間・約 70 ドキュメント・プロトタイプ 0」の停滞を生んだ。本スキルは導出を自走化してこれを解く。
- **踏み込まない**: 技術スタック・データモデル・アーキテクチャ・インフラは下流 `tech-designer`（G3）の責務。本スキルはフロントエンドの体験・UX に必要な corpus までを導出する（Claude Design がフロントエンド専用であることに整合）。

## いつ使う / 使わない

**使う**:
- アンカー4点が揃っており（あるいは PRFAQ・デザイン原則の核がある）、プロトタイプの設計を一気に固めたいとき
- 「Claude Design 用のブリーフを作って」「企画から画面まで導出して」と言われたとき
- 旧 service-designer/prototype-designer の出力を作り直したいとき

**使わない**:
- アンカー自体がまだ固まっていない（PRFAQ がターゲットと中核シナリオを名指せていない）→ 先にアンカーを固める（G1 / `service-designer` の企画部分）
- 技術設計・実装・分解の段（→ tech-designer / issue-decomposer / feature-team）

## 入力（アンカー4点）

ADR-0002 のアンカー集合。人間の意思決定境界であり、これより下流は本スキルが導出する。

| # | アンカー | 内容 | DoD（これを満たさなければ導出に進まない） |
|---|---|---|---|
| ① | PRFAQ | 価値とゴール。対外公式ストーリー（PR）＋ FAQ ＋ ゴールライン（提供する/しない） | **ターゲットと中核シナリオを鋭く名指している**こと（薄い PRFAQ は的ズレを全派生に伝播させる） |
| ② | デザインプリンシプル | 優先順位の真実源。各原則は「A vs B → A を優先」＋理由＋守る相手のトレードオフ形 | 主要なトレードオフが言語化されている |
| ③ | 提供形態 | iOS / Android / Web 等。技術スタックは含めない | 第一形態が決まっている |
| ④ | マネタイズ | する/しないの二値（＋方針）。無料/有料の境界は「繰り越し決定」でよい | 二値が決まっている |

**アンカーの所在**（ADR-0010）:
- 人間のアンカー執筆面は Obsidian vault（任意）。corpus of record は repo 側。
- 本スキルは起動時にアンカーの所在を確定する。優先順位: (a) 起動引数で渡されたパス → (b) カレント repo の `docs/design/anchors/` → (c) Obsidian vault（`mcp__claude_ai_Obsidian_MCP__*` で読む。場所が不明なら **一度だけ** 確認してよい＝入力の所在特定は G1 の延長）。
- 補助入力（任意）: 参考リポジトリ、既存の導出物（旧 service-designer 出力等）。これらは self-grill の **答え合わせ材料（corpus）** として使うが、丸写ししない。

## 出力（repo-native corpus + ブリーフ）

ADR-0010: corpus of record は **カレントのサービス repo の `docs/design/` サブツリー**に repo-native Markdown で置く。コードと原子的にバージョン管理し、PR 同梱・ツール非依存・並列デュアル訂正（ADR-0006）と整合させる。

```
<service-repo>/docs/design/
  00-overview.md            … 導出の起点・サービス一行説明・アンカー要約・このバンドルの読み方
  anchors/                  … アンカーのスナップショット（vault から取り込んだ場合。導出の安定入力）
    prfaq.md  principles.md  provider-and-monetization.md
  10-persona.md             … ペルソナ（中心＋周辺層。PRFAQ から導出）
  20-usage-scenes.md        … 使用シーン（中核の反復体験を時系列で・エッジ含む）
  30-feature-scope.md       … 機能スコープ（提供する/しない・課金境界を機能粒度で）
  40-design-concept.md      … デザインコンセプト（原則を視覚トーンに翻訳）
  50-screens/               … 全画面仕様（端折らない・1 画面 1 ファイル）
    _index.md               … 画面一覧と機能スコープへの被覆マップ
    <area>/<screen>.md      … 各画面（目的・主要素・状態・遷移・繰り越し論点）
  90-claude-design-brief.md … Claude Design に渡すプロトタイプブリーフ（中核成果物）
  DECISIONS.md              … 決定ログ＝正当化トレイル（self-grill の自答結果を全記録）
  OPEN-QUESTIONS.md         … 繰り越し決定・G2 で人間が確定する未決の論点一覧
```

- コミットは `docs(design):` スコープで分離可能にする（`git log -- docs/design/` でフィルタ）。
- corpus とコードは履歴が混在するが受容する（ADR-0010）。

## 自走規律（停止せず・決めて・記録して・進む）

ADR-0004 / 0005 / 0007 の中核。**曖昧点で停止して人間に一問一答しない**。

- **decide-record-proceed**: 内部の曖昧点は、根拠ある決定を下し → `DECISIONS.md` に記録し → 進む。記録が正当化トレイルを兼ねる。
- **停止してよいのは2種だけ**:
  - (A) 設計されたゲート: アンカーの所在特定（入力境界）と、成果物を **G2（プロトタイプゲート）** に引き渡す瞬間。
  - (B) 不可逆な対外書き込み: git push / PR merge / deploy / 課金 / 外部送信。**本スキルはこれらを行わない**（repo への commit は可逆・内部的なので自律）。
- **self-grill（ADR-0005）**: griller と answerer の両役を自分で演じ、アンカー／既存 corpus／参考リポを答え合わせ材料に **自答** して要件を精緻化する。人間対話の grill を置き換える。
- **ティアリング（決定単位ランタイム判断）**: すべてを self-grill し、個々の決定が「**context から既約 かつ 不可逆/高ステークス**」のときだけ G2/`OPEN-QUESTIONS.md` で晒す。それ以外は decide-record-proceed で確定する。
- **設計そのものを変える判断**（ハーネスの ADR を覆す類）が要るときだけ、例外的に人間に確認する。

## 手順

### Phase 0 — 入力確定（境界）
1. アンカーの所在を上記優先順位で特定。Obsidian vault から取り込む場合は `docs/design/anchors/` にスナップショットを書き出す（安定した repo-native 入力にする）。
2. アンカー DoD（上表）をチェック。**満たさないアンカーがあれば導出に進まず、不足を具体的に指摘して G1 へ差し戻す**（ここは停止してよい＝入力境界）。
3. 出力先 `docs/design/` を用意し、`00-overview.md` と空の `DECISIONS.md` / `OPEN-QUESTIONS.md` を初期化。

### Phase 1 — foundation 導出（直列・1 体）
`service-deriver` エージェントを **slice=foundation** で起動し、相互依存する4つを一貫性をもって導出させる（各記述は対象サービスのアンカーに既約。socialcoffeenote は例示）:
- `10-persona.md` … 中心ペルソナ＋周辺層（デザイン原則が優先順位を定めるならそれに従う）。jobs-to-be-done を中核の反復体験（コアループ）に接続。
- `20-usage-scenes.md` … コアループ（PRFAQ が定める中核の反復体験）を時系列で具体化。入口設計（未ログイン/未登録からの価値体験→転換）、デザイン原則の安全要求に沿う事故・不可逆防止などエッジを含む。
- `30-feature-scope.md` … 提供する/しない（PRFAQ ゴールライン）に整合。アンカー④とデザイン原則の無料/有料線引き・禁じ手を機能粒度で割り当て、不可侵領域を侵さない。原則が排除する種類の機能を明示的に外す。
- `40-design-concept.md` … デザイン原則を視覚トーン（色・タイポ・密度・モーション・コピーの語り口）に翻訳。原則由来の構造制約を含む。

foundation の self-grill 決定はすべて `DECISIONS.md` に追記させる。

### Phase 2 — 全画面仕様 導出（並列・複数体）
1. `30-feature-scope.md` から画面を列挙し、`50-screens/_index.md`（画面一覧＋機能被覆マップ）を作る。**機能スコープの各機能が最低1画面に被覆されること**を確認（被覆漏れは端折りの兆候）。
2. 画面を機能エリア（例: records / social / foundation）でグルーピングし、エリアごとに `service-deriver` を **slice=screens, area=<area>** で **並列起動**（各エリアは別ファイル群に書くので write 競合しない）。各画面ファイルは目的・主要素・全状態（empty / loading / error / 未ログイン / 課金壁）・遷移・繰り越し論点を持つ。
3. 各並列ジョブの決定を `DECISIONS.md` に追記させ、オーケストレーター（このスキル）が統合・重複排除する。

### Phase 3 — Claude Design ブリーフ生成（直列・このスキル本体）
`90-claude-design-brief.md` を生成。Claude Design が **完全で・お粗末でない** プロトタイプを一発で起こせる密度にする（後述「ブリーフの DoD」）。

### Phase 4 — セルフレビュー（DoD 検証）→ G2 引き渡し
- 「品質バー / DoD」チェックリストを全項目検証。未達があれば該当 slice を再導出（停止せず自走で埋める）。
- `git status` を確認し、`docs(design):` で commit。
- **G2 へ引き渡す**（停止してよい点）: 人間に「ブリーフを Claude Design に渡す → プロトタイプ → ゲートでレビュー（ADR-0006 の並列デュアル訂正）」を促す。`OPEN-QUESTIONS.md` の未決論点を提示する。

## self-grill 観点リスト（artifact ごと）

各 artifact を導出するとき、answerer の草案に対して griller がこの観点で詰め、`DECISIONS.md` に「論点 → 自答 → 根拠（どのアンカー/原則に既約か、または繰り越し）」を残す。観点は **サービス非依存**。socialcoffeenote は走例として「例:」で示すだけで、規則そのものはアンカー参照で書く。

> このスキルは任意のサービスに適用する。観点の操作部は常に「対象サービスのアンカー④点が定めたことに整合するか」で判定する。サービス固有のビジネスルール（無料/有料の線引き・不採用機能の種類等）を普遍則として埋め込まない。

**ペルソナ（10）**
- PRFAQ が名指すターゲットと中核シナリオから **既約に** 導けているか（憶測のペルソナ造形をしていないか）。
- デザイン原則が中心ペルソナ/優先順位を定めているなら、中心と周辺層を分離してそれに一致させているか（例: socialcoffeenote は中心 N=1・周辺 ROM 層）。
- ペルソナの jobs-to-be-done が、PRFAQ の中核シナリオ（＝そのサービスのコアループ）と一致するか。

**使用シーン（20）**
- そのサービスのコアループ（PRFAQ が定める中核の反復体験）が、全周にわたり時系列のシーンとして立っているか。
- デザイン原則が要求する体験品質をシーンが満たすか（例: socialcoffeenote の原則2「淹れながら一瞬で書ける・書きかけが消えない」）。
- 入口設計（未ログイン/未登録からの価値体験 → 転換）が提供形態に沿って表現されているか。
- 事故・不可逆を防ぐシーンが、デザイン原則の安全要求に沿って含まれているか（例: socialcoffeenote の原則4「公開範囲は保存時必須選択」）。

**機能スコープ（30）**
- PRFAQ の「提供する/提供しない」と一字一句整合するか。将来も提供しない機能を誤って含めていないか。
- デザイン原則が排除する種類の機能を機能集合から外しているか（例: socialcoffeenote は点数・ランキング・ゲーミフィケーション・コメントを不採用）。
- アンカー④（マネタイズ）とデザイン原則が定めた無料/有料の線引き・禁じ手を、機能粒度で正しく割り当てているか。アンカーが侵してはならない領域を定めているなら、それを侵していないか（例: socialcoffeenote はコアループへの課金・上限が禁じ手）。
- 触ってから決めるのが妥当な繰り越し決定（例: 無料/有料境界の細部）は、勝手に固めず `OPEN-QUESTIONS.md` に未決として出しているか。

**全画面仕様（50）**
- 機能スコープの各機能が画面に **被覆** されているか（被覆マップで漏れゼロを確認）。端折りは品質バー違反。
- 各画面が起こり得る全状態（empty / loading / error / 未ログイン or 未認可 / 課金壁 / 保存中・下書き 等、そのサービスに該当するもの）を持つか。
- 繰り越し決定をプロトタイプ上で **触れる/決められる** 形に表現しているか（ADR-0002: プロトタイプは未決論点を表現でき、人間が決定を下せる必要がある）。
- 入力/保存系画面が、デザイン原則の安全要求を満たすか（例: socialcoffeenote は公開範囲を保存時に毎回必須選択・デフォルトなし）。
- 課金や注目誘導の見せ方が、デザイン原則のトーン要求と擦れていないか（例: socialcoffeenote の原則1「落ち着いた空気・課金が注目を独占しない」）。

**デザインコンセプト（40）**
- 各デザイン原則が具体の視覚決定（パレット・タイポ・余白/密度・モーションの有無と強度・コピーのトーン）に翻訳されているか。抽象語で止めず、実装可能な指定まで降ろしているか。
- 一覧やフィードの構造が、デザイン原則に反する偏り（例: 課金コンテンツが構造的に目立つ）を生んでいないか。

**ブリーフ（90）**
- 上記すべての要約が、Claude Design 単体で完全なプロトタイプを起こせる密度で入っているか（後述）。

## 品質バー / DoD（端折り禁止）

「minimal」は **人間の入力・対話の最小化** を指し、導出物の最小化ではない（CONTEXT.md）。以下を全部満たして初めて G2 へ引き渡せる。

- [ ] アンカー4点を読み、DoD を満たすことを確認した（不足は G1 差し戻し）。
- [ ] ペルソナ・使用シーン・機能スコープ・デザインコンセプトが揃い、相互に矛盾しない。
- [ ] **全画面**が `50-screens/` に1画面1ファイルで存在し、機能被覆マップに漏れがない（「主要画面だけ」は不可）。
- [ ] 各画面が全状態と遷移を持つ。
- [ ] マネタイズの割り当てが、アンカー④とデザイン原則が定めた線引き・禁じ手に整合している（アンカーが侵してはならない領域を定めているならそれを侵していない）。
- [ ] 繰り越し決定が `OPEN-QUESTIONS.md` に未決として並び、勝手に固めていない。
- [ ] `DECISIONS.md` が **読める**正当化トレイルになっている（論点→自答→根拠が追える）。
- [ ] `90-claude-design-brief.md` がブリーフ DoD を満たす。
- [ ] `docs(design):` で commit 済み（`git status` クリーン）。

### ブリーフの DoD（Claude Design がお粗末でないプロトタイプを一発で起こせる）
`90-claude-design-brief.md` は次を含む:
1. サービス一行説明・トーン（誰の何をどう解くか・空気感）。
2. 対象ペルソナと中核シナリオ（プロトタイプで再現すべき体験の筋）。
3. デザインコンセプト: パレット・タイポ・密度・モーション・コピーのトーンを実装指定レベルで。
4. 画面一覧と、各画面の目的・主要素・**再現すべき状態**（empty/error/未ログイン/課金壁等）。
5. 繰り越し決定（未決論点）を **プロトタイプ上で触れる形に** という明示指示（例: 無料/有料境界をトグルで体験できるように）。
6. スタック前提: フロントエンドのみ（Claude Design はバックエンド生成なし）、提供形態に沿うフォーム（iOS なら mobile viewport 等）。
7. ハンドオフ DoD: 各状態のスクショ・コンポーネント spec（機械可読）・デザイントークン・README（stack/パス/DoD）を出し、Claude Code へハンドオフバンドルとして戻すこと（このバンドルが画面仕様＝正当化トレイルを兼ねる。ADR-0003）。

## ゲートと不可逆操作

- 本スキルが人間に晒すのは **G2（プロトタイプゲート）** のみ。G2 では人間が Claude Design（claude.ai）と Claude Code を並列稼働させ、docs 修正（Claude Code）とプロトタイプ修正（Claude Design）を同時に出す（ADR-0006 並列デュアル訂正）。本スキルは G2 で人間が docs を直接編集できるよう、corpus を repo-native に保つ。
- 本スキルは push / PR / merge / deploy / 課金 / 外部送信を行わない。commit までが自律範囲。
